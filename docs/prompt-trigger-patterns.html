好 prompt 的关键是“触发正确动作”

你不需要控制每个细节，但要告诉 Codex：先探索还是直接改、是否联网、是否需要浏览器/PDF 验收、是否要沉淀成 skill、是否允许并行 agent。

目录

先探索再修改

适用

代码仓库不熟、论文口径复杂、你担心它改错文件。

请先用只读方式探索：读项目入口、相关配置、最近提交和可能受影响文件。
确认当前实现形状后，再给出最小改动方案并实施。
不要在探索阶段改文件。

联网查官方来源

适用

API、模型、法律、价格、规则、产品能力这类会变的信息。

这件事可能已经变化，请联网查证。
优先官方文档或一手来源；回答里给出链接和日期/版本信息。
如果官方文档和本仓库实现冲突，请指出冲突，不要直接改。

触发 skill

显式触发

请使用 $pdf 检查这个论文 PDF 的版式和关键页。
请使用 $skill-creator，把这套重复工作流沉淀成一个 skill。

隐式触发

请把这段对话整理成结构化 Notion 决策记录。
请用 Playwright 打开本地页面并截图验收。

显式触发适合你很确定要用哪个 skill；隐式触发适合你只描述任务，让 Codex 根据 skill description 自己匹配。

要求真实验收

代码

改完请运行相关测试或静态检查。
如果不能运行，请说明原因、缺口和替代验证。

网页/HTML

请用 Playwright 在桌面和移动视口打开页面。
保存截图，检查 console 是否有 error/warning，并确认没有横向溢出。

论文/PDF

请编译 LaTeX 或读取现有 PDF。
抽取文本检查术语和指标，再渲染关键页检查图表与版式。

要求可恢复长任务

这是长跑批任务，请设计为可恢复：
- 结果 append 写入；
- 启动时跳过已完成组合键；
- 保存 manifest/status/logs；
- API 错误做指数退避；
- 汇总前检查失败和缺失行。

请求 subagents

这个任务可以并行。请 spawn 3 个 subagents：
1. explorer：只读检查代码结构和风险；
2. explorer：只读检查论文事实和术语；
3. worker：在不重叠文件范围内实现图表同步。
最后汇总每个 agent 的结论、变更文件和剩余风险。

只在并行收益明显时要求 subagents。它们会消耗更多 token 和时间；并行写同一文件容易冲突。
