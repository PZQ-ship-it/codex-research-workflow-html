核心判断

只问模型

适合稳定知识、概念解释、帮你拆问题。不要让它凭记忆给最新事实、价格、规则、论文细节。

让模型调工具

适合需要读仓库、跑命令、联网查证、看 PDF、截图验收、改文件的任务。prompt 里要点名证据。

沉淀成 skill

适合重复出现、有固定顺序、容易出错的工作流，例如终稿审阅、图表同步、可恢复跑批。

一个实用路由口诀

  1. 事实会变吗？会变就要求联网或读项目事实源。
  2. 答案依赖仓库吗？依赖就先让 Codex 用 rg / shell 读代码。
  3. 会改文件吗？要求“先说明改动点，再小范围补丁修改，最后跑验证”。
  4. 有视觉/版式吗？要求 Playwright、PDF 渲染或截图证据。
  5. 任务会重复吗？要求用 $skill-creator 沉淀成 skill。
  6. 任务可并行吗？明确要求 spawn subagents，否则 Codex 通常不应自行派生。

对你的项目来说，最常用的组合是：先读 CODEX.md 和事实源，再跑实验或生成图表，最后用 PDF/Playwright 级验收把结果关上。
