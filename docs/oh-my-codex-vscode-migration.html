不要照搬 OMX runtime，把可复用工作流迁移成 Codex 原生能力

oh-my-codex 很有价值，但它的强项是 Codex CLI 上的多 agent 编排、tmux/runtime 状态和 hooks。Windows + VS Code 场景下，第一版最稳的做法是只迁移思想和 skill 表面：让 Codex IDE 能用 $codex-... 直接触发。

npm versionoh-my-codex@0.18.1
本地研究副本.codex-local\oh-my-codex\repo
迁移策略VS Code native v1

目录

结论

可以迁移一部分，而且值得迁移；但不要在原生 Windows 的 VS Code 交互里直接依赖 omx setup、tmux team runtime、.omx/state 或 OMX hooks。第一版迁移目标是：

  • 把成熟的工作流压缩成 Codex skills，放在本仓库 skills/ 作为 canonical source。
  • 验证后复制到 C:\Users\Administrator\.codex\skills，让 VS Code Codex 聊天里可以显式调用。
  • 把人类需要理解的边界、安装风险和使用方式写成 HTML 文档，而不是藏在 prompt 里。

本次没有运行 npm install -g oh-my-codex、omx setup、omx update，也没有安装 OMX hook。upstream 只作为静态研究来源。

迁移矩阵

OMX 能力 迁移判断 VS Code Codex 落地方式 注意事项
$deep-interview 直接迁移 $codex-deep-interview：先查事实，再一问一答澄清偏好，最后给 handoff brief。 去掉 omx question 和 .omx/specs 强依赖。
$ralplan / consensus plan 直接迁移 $codex-consensus-plan：输出 decision-complete plan、测试和假设。 适合你常用的“先计划，再实现”。
$ralph 直接迁移 $codex-completion-loop：实现、验证、修复、交付证据。 不使用 OMX 持久状态；依赖当前 Codex 会话和本地验证。
$visual-ralph 直接迁移 $codex-visual-acceptance：Playwright/PDF/screenshot 验收 UI、HTML、图表。 非常适合你当前 HTML 笔记和论文作图工作流。
$ultraqa / code review 直接迁移 $codex-adversarial-qa：用 hostile scenarios 检查代码、论文、图表、跑批。 报告必须带证据，不把猜测当 finding。
$design 直接迁移 $codex-design-brief：维护轻量设计/决策源。 可与 Figma skill 或 HTML notes 并用。
$team 改造迁移 $codex-native-subagent-team：在用户明确要求 subagents/parallel agents 时，拆成 native subagent lanes。 不搬 tmux panes、team state、worker mailbox；并行只在明确有收益时使用。

omx setup / native hooks / .omx/state v1 不迁移 不修改 Codex hooks，不写 OMX runtime 状态。 避免与现有 C:\Users\Administrator\.codex 配置互相覆盖。

新增 VS Code 原生 skills

本次新增 7 个不与 upstream 冲突的 skill 名称，canonical source 位于本仓库 skills/，并在验证后复制到全局 Codex skills 目录：

  • $codex-deep-interview：需求澄清，先探索事实，再问偏好。
  • $codex-consensus-plan：生成可直接执行的计划。
  • $codex-completion-loop：执行到有证据的完成状态。
  • $codex-visual-acceptance：浏览器/PDF/截图级视觉验收。
  • $codex-adversarial-qa：用反例和极端场景压测结果。
  • $codex-native-subagent-team：把 OMX $team 改造成显式 native subagent 并行协作。
  • $codex-design-brief：沉淀轻量设计与决策源。

在 VS Code Codex 中可以这样触发：

$codex-deep-interview "帮我澄清这个论文图表重做任务，先不要改文件"

$codex-consensus-plan "为 HTML 工作流仓库增加一页高级 Codex runtime 模式说明"

$codex-completion-loop "按已批准计划实现并给出验证证据"

$codex-visual-acceptance "检查这个 HTML 页面桌面和移动端是否有布局问题"

$codex-adversarial-qa "审查这次论文 fact sync 是否还有术语和指标不一致"

$codex-native-subagent-team "这个任务可以并行，请拆出主线程 critical path 和两个只读 explorer lane"

$codex-design-brief "为这个研究笔记网站补一个轻量 DESIGN 决策源"

为什么不直接完整安装 OMX

OMX 的 README 明确说它主要为 macOS/Linux + Codex CLI 调优，原生 Windows 和 Codex App 不是默认体验；同时它的 team runtime 强依赖 tmux/psmux、状态目录、hooks 和长期运行的终端环境。这些能力很强，但不适合作为你当前 Windows VS Code Codex 交互的默认底座。

官方 Codex 文档提供了更稳的原生入口：IDE extension、CLI、skills、plugins、hooks、AGENTS.md 和 MCP。v1 选择 skills + HTML playbook，是因为这两者最容易被 VS Code 交互界面直接使用，也最容易复制到别的科研项目。

后续可选升级

  1. 如果你想试完整 OMX runtime，在 WSL2 中单独安装和测试，不污染 Windows 主 Codex 配置。
  2. 如果这 6 个 skill 稳定，可以把它们打包成 repo-local Codex plugin，方便一键分发。
  3. 如果某个 workflow 反复扩张，再拆出专门 HTML 页面或更细的 skill。
